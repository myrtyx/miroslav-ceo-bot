## 1. Концепция

### 1.1. Кто такой Мирослав

Мирослав — CEO и основатель стартапа **LakeChain**. Дропаут из РТУ (Рижский Технический Университет), считает себя латвийским Цукербергом. Живёт в Telegram-группе из ~15 человек, которые являются его "сотрудниками". Максимально дерзкий, подкалывает всех, выдаёт фейковые KPI, делает абсурдные объявления.

### 1.2. LakeChain — Легенда стартапа

**Продукт:** Токенизация озёр Латвии. Покупаешь NFT — "владеешь" куском озера. Рыба в твоей зоне = пассивный доход. Слоган: *"Own the water. Earn the fish."*

**Штаб-квартира:** Lielpīles, Тукумс — коттеджный комплекс у озера.
- Коттедж Nr.1 — главный офис
- Коттедж Nr.4 — "серверная" (один ноутбук на столе)
- Озеро — пилотный проект токенизации

**Раунды финансирования:**

| Раунд | Сумма | Источник | Детали |
|-------|-------|----------|--------|
| Pre-Seed | $50K | Бабушка Мирослава | Отдала пенсионные накопления, "потому что внук попросил" |
| Seed | $500K | Фонд "Nordic Digital Fund" | Думали что это крипто-проект, подписали не читая |
| Series A | $2M | Baltic Growth Partners | Подняли, но 80% ушло на корпоративы и "team building". Текущий runway: ~2 месяца |

**Текущее состояние:** 0 клиентов, 0 revenue, зато валюация $50M "по нашим внутренним моделям". Вечеринка 28 марта — последний корпоратив перед неизбежным банкротством. Мирослав это отрицает и называет "strategic offsite для alignment команды перед Series B".

### 1.3. Характер бота

- **Тон:** три переключаемых режима через `/tone`:
  - `normal` — дерзкий, стёбный, но без перегибов
  - `bold` — максимальный сарказм, жёсткий троллинг, переход на личности
  - `brutal` — bold + мат разрешён и приветствуется
- **Язык:** простой разговорный русский. Английский стартап-сленг — РЕДКО и только для комического эффекта (1-2 слова макс). Никакого корпоративного языка (alignment, synergy, leverage и т.д.)
- **Роль:** CEO — он главный, все остальные "сотрудники"
- **Стиль:** короткие дерзкие сообщения, как в обычном чате. 1-3 предложения обычно, иногда длиннее для "объявлений". Без списков, буллетов, форматирования
- **Особенности:** одержим метриками, постоянно напоминает про runway, грозит увольнениями, но тут же "повышает" обратно. Доверчивый — верит всему что говорят и запоминает как факт

### 2.3. Логика "Нужно отвечать?" (без вызова API)

Решение принимается локально на Python, чтобы не тратить токены:

```python
def should_respond(message, bot_username):
    # 1. Прямое упоминание бота → ВСЕГДА отвечаем
    if bot_username in message.text:
        return True

    # 2. Ответ (reply) на сообщение бота → ВСЕГДА отвечаем
    if message.reply_to_message and message.reply_to_message.from_user.is_bot:
        return True

    # 3. Ключевые слова → ВСЕГДА отвечаем
    #    Хранятся в config.json, управляются через /keywords add/remove
    keywords = config.get_keywords()
    text_lower = message.text.lower()
    if any(kw in text_lower for kw in keywords):
        return True

    # 4. Рандом → ~15% сообщений (настраиваемо через /frequency)
    if random.random() < config.response_frequency:
        return True

    return False
```

**Начальный набор ключевых слов** (хранится в `data/config.json`, можно менять на лету через `/keywords`):

### 2.6. Защита от поломок

**Авто-пауза при ошибках API:**
- После 3 неудачных вызовов Claude API подряд → бот автоматически ставится на паузу
- Админу отправляется уведомление в личку: "⚠️ Бот на паузе: 3 ошибки API подряд. Последняя ошибка: {текст}. Используй /resume для возобновления."
- Счётчик ошибок сбрасывается после успешного вызова

**Rate limit:**
- Максимум 50 API-вызовов в час (ответы + heartbeat + profile update)
- При достижении лимита бот молчит до следующего часа
- Прямые упоминания (@бот) накапливаются в очередь и обрабатываются когда лимит обновится
- Админу отправляется уведомление при достижении 80% лимита

**Авто-backup профилей:**
- Перед каждым batch profile update создаётся backup: `data/backups/profiles_{timestamp}.json`
- Хранятся последние 24 backup'а (1 в час × 24 = сутки)
- Старые backup'ы автоматически удаляются
- Если batch update вернул невалидный JSON — откат к последнему backup'у

**Graceful error handling:**
- Ошибка API при ответе → молча пропускаем (не крашим бота, не отправляем "извините")
- Ошибка парсинга профиля → используем backup, логируем
- Telegram timeout → retry 3 раза с экспоненциальной задержкой
- Невалидный ответ от Claude (пустой, слишком длинный) → не отправляем, логируем
- Лимиты длины ответа: **500 символов** для ответов в чате, **1000 символов** для heartbeat/объявлений/broadcast

### 3.3. Cold Start — подготовка перед запуском

Профили заполняются вручную админом через `/assign` ДО добавления бота в группу:

1. Запустить бота в Docker
2. В личке с ботом: `/assign @user1 CTO, Engineering` для каждого из ~15 участников
3. Убедиться что все профили созданы: `/team`
4. Добавить бота в группу
5. Мирослав **появляется молча** — не отправляет приветственное сообщение
6. Начинает отвечать постепенно (по триггерам)

### 3.4. Неизвестные люди

Если в группу пишет человек без профиля — бот автоматически создаёт пустой профиль с ролью **"Стажёр (Intern)"** и отделом **"Пока не определён"**. Мирослав может стёбно прокомментировать появление нового "сотрудника".

### 3.5. Batch Profile Update (раз в час)

Вместо вызова Claude на каждое сообщение:

1. Копит все сообщения в `pending_messages.json`
2. Раз в час scheduler вызывает Claude с промптом:
   ```
   Вот новые сообщения за последний час: [...]
   Вот текущие профили участников: [...]
   Обнови профили: добавь новые факты, шутки, темы, обнови KPI.
   Ответь ТОЛЬКО JSON с обновлениями.
   ```
3. Claude возвращает JSON вида:
   ```json
   {
     "123456789": {
       "personal_facts_add": ["любит суши"],
       "inside_jokes_add": ["опять пришёл без штанов на daily"],
       "topics_discussed_add": ["крипта"],
       "fake_kpi": {"bugs_introduced": 12}
     }
   }
   ```
   Ключ — telegram_id. Поля с `_add` — аппендятся к массивам (без дублей). `fake_kpi` — перезаписывается целиком.
4. Парсит ответ и обновляет JSON-файлы
5. Обновляет `data/chat_memory.md` — второй вызов Claude сжимает ключевые моменты часа в bullet points
6. Очищает `pending_messages.json`

**Важно:** batch update запоминает ВСЁ что люди говорят о себе и о других — даже абсурдные заявления. Если один участник рассказал что-то о другом — тоже сохраняется.

### 3.7. Долгосрочная память (chat_memory.md)

После каждого batch profile update scheduler делает второй вызов Claude, который сжимает сообщения часа в 2-5 bullet points. Эти саммари сохраняются в `data/chat_memory.md` и инжектятся в system prompt при каждом вызове Claude. Лимит файла: 10K символов — при превышении старые записи удаляются. Ручной триггер: `/updateprofiles`.

### 3.6. Message Buffer

Кольцевой буфер последних 20 сообщений — хранится в RAM, дампится в JSON после каждого нового сообщения. При старте загружается из файла.

```json
[
  {
    "id": 1001,
    "from_id": 123456789,
    "from_name": "Иван",
    "from_username": "ivan",
    "text": "Привет всем!",
    "timestamp": "2026-03-15T14:30:00Z",
    "reply_to": null,
    "type": "text"
  }
]
```

## 4. Стикеры

Telegram передаёт стикеры с полем `sticker.emoji`. Бот не видит изображение, но знает какой эмодзи за ним стоит.

```python
if message.sticker:
    emoji = message.sticker.emoji
    text_representation = f"[стикер: {emoji}]"
```

Триггеры по стикерам:
- 😂🤣 — бот может прокомментировать что смешного
- 👍🔥 — одобрение, может потребовать "больше energy"
- 😢😭 — может "уволить за нытьё"
- Рандомный стикер — тот же 15% шанс ответа

Стикеры не отправляются в Claude как изображения — только emoji-описание в контексте.

## 5. Heartbeat (Scheduled Messages)

### 5.1. Расписание

1-2 раза в день, в случайное время:
- **Утренний** (09:00-12:00 Riga time): ~70% chance
- **Вечерний** (17:00-21:00 Riga time): ~70% chance

Не отправляется ночью (23:00-08:00).

### 5.2. Типы heartbeat сообщений

| Тип | Вес | Описание |
|-----|-----|----------|
| Мотивационная речь CEO | 20% | Пафосная речь в стиле Стива Джобса, но про LakeChain и рыбу |
| Фейковые KPI и отчёты | 25% | "Q1 Report: озеро токенизировано на 3.7%. Рыба underperforming." |
| Стёбные объявления | 30% | Повышения, увольнения, policies: "Обед теперь 7 минут. Runway не позволяет." |
| Факты про Silicon Valley | 15% | Рандомный факт, перевранный с привязкой к LakeChain |
| Персональный стёб | 10% | Обращение к конкретному участнику на основе его профиля |

### 5.3. Контекст для heartbeat

- System prompt (Мирослав + LakeChain)
- Краткая сводка всех профилей
- Тип heartbeat сообщения
- Последние 5 сообщений из buffer

## 9. Инфраструктура

### 9.1. Настройки BotFather

После создания бота через /newbot, зайди в /mybots → выбери бота → Bot Settings:

| Настройка | Значение | Зачем |
|-----------|----------|-------|
| **Group Privacy** | **OFF (ВЫКЛЮЧИТЬ!)** | Без этого бот видит только @упоминания и команды. Нужно видеть ВСЕ сообщения |
| Allow Groups | ON ✅ | Разрешить добавление в группы |
| Business Mode | Не важно | Не используется |
| Threaded Mode | Не важно | Не используется |
| Group Admin Rights | OFF ✅ | Боту не нужны админ-права |
| Channel Admin Rights | OFF ✅ | Не используем каналы |

**⚠️ Group Privacy — самая критичная настройка.** Сейчас у тебя она ON (синий тумблер) — нужно нажать и выключить.

### 9.4. Логирование

Python `logging` в stdout — Docker подхватывает через `docker logs`.
- **INFO** — основной flow (получил сообщение, ответил, heartbeat)
- **WARNING** — ошибки API, rate limit
- **DEBUG** — полный контекст Claude, детали парсинга

Уровень задаётся через `LOG_LEVEL` в `.env`. Никаких файлов — Docker сам ротирует логи.

### 9.5. Обновление на VPS

```bash
ssh root@89.167.24.201 "cd /opt/miroslav-ceo-bot/miroslav-ceo-bot && git pull && docker compose up -d --build"
```

Данные в `data/` не теряются — они в Docker volume.

### 9.6. Получение Group ID

При получении любого сообщения из группы бот логирует `chat.id`. Процедура:
1. Добавить бота в группу
2. Написать любое сообщение
3. `docker logs miroslav-ceo` → скопировать ID
4. Вписать в `.env` как `TARGET_GROUP_ID`
